# Todo管理Webアプリ 仕様書

## 1. 概要

就業時間に合わせてTodoの入力・進捗管理・確認を促す、個人向けのWebアプリケーション。
朝の就業開始前にTodoを登録し、就業終了時にステータス(未着手/完了/継続)の入力漏れをChrome通知でチェックする。

本書は `docs/requirements.md` に記載された要求を、利用者・スコープ・受入れ基準・異常系の観点で具体化したものである。

### 1.1 前提条件・用語

- **「当日」の判定基準**: 端末のブラウザのローカル日付(0:00〜23:59)を基準とする。ローカル日付が変わった時点で新しい日のTodoリストとして扱う(前日以前のTodoの繰越・保持方法は6章の未確定事項を参照)。
- **時刻判定の粒度**: 就業開始/終了時刻との比較は分単位(HH:mm)で行う。設定時刻と現在時刻の「時」「分」が一致した時点を発火タイミングとする。
- **通知の発火回数**: 就業開始通知・就業終了通知は、それぞれ1日1回のみ発火する。一度発火した通知は、同日中にページを再読み込み・再訪問しても再発火しない。
- **就業開始通知の発火条件**: 就業開始時刻の時点でTodoが1件も入力されていなくても、既に入力済みであっても、就業開始通知は必ず1回発火する(定時リマインダーとして扱うため、入力状況による発火有無の判定は行わない)。

## 2. 利用者

| 項目 | 内容 |
|---|---|
| 想定利用者 | Webアプリ利用者本人(1名)。日々の業務Todoを自己管理する個人ユーザーを想定 |
| 利用端末 | Chromeブラウザが動作するPC(業務開始・終了時にブラウザ/アプリを開いている状態) |
| 利用場面 | 平日の就業開始前・就業終了時 |
| 前提知識 | 特別な知識は不要。Todoの入力とステータス選択ができれば利用可能 |

マルチユーザー・チーム共有・他者への通知配信は想定しない(後述スコープ外)。

## 3. スコープ

### 3.1 対象範囲(In Scope)

1. **Todo入力**: 就業開始前に、当日実施するTodoをテキストで複数件入力できる。
2. **就業開始通知**: あらかじめ設定した就業開始時刻になったら、Chromeの通知機能でユーザーに知らせる(Todo入力を促す)。
3. **ステータス管理**: 登録直後のTodoは内部状態「未設定」を持ち、利用者は以下いずれかのステータスを明示的に設定できる。
   - 未着手
   - 完了
   - 継続

   (「未設定」はユーザーが選択する値ではなく、未入力を検知するための内部状態。詳細は4章の用語定義を参照)
4. **就業終了チェック・通知**: あらかじめ設定した就業終了時刻になったら、その日の全Todoに対してステータスが設定済みかを自動チェックし、未設定のTodoがあればChrome通知で知らせる。

### 3.2 対象外(Out of Scope)

- 複数ユーザーでのアカウント管理・ログイン認証・権限管理
- Todoの他者への共有・チーム機能
- 就業開始/終了時刻の自動判定(カレンダー連携、勤怠システム連携等)。時刻は利用者が設定する前提とする
- モバイルアプリ(ネイティブ)対応。本仕様はWebアプリ(Chromeブラウザ)を対象とする
- Todoの繰り返し設定・リマインダーの詳細カスタマイズ(スヌーズ等)
- 音声・メール・Slack等、Chrome通知以外の通知チャネル
- ブラウザ/端末がスリープ・シャットダウン・非起動の場合の通知配信保証
- Todoの分析・レポーティング機能(週次/月次集計等)
- オフライン動作の保証
- 登録済みTodoの文言編集・削除(誤入力の訂正は本バージョンでは非対応。登録前に入力欄で内容を確認する運用とする)
- 通知が届かなかった場合(タブが閉じていた、PCがスリープ状態だった等)の代替表示・再通知機能
- 同一アプリを複数タブ/複数ウィンドウで同時に開いた場合の通知重複排除

## 4. 機能要件と受入れ基準

以下は Given(前提)/ When(操作・イベント)/ Then(期待結果) 形式で記述する。各Thenは実装時にそのまま自動テスト・手動テストの検証項目として使用できるよう、具体的な値・文言・件数で記述する。

> **用語定義(テスト可否のための明確化)**: Todoのステータスには、ユーザーが選択可能な3値「未着手」「完了」「継続」に加え、登録直後で一度もステータス操作をしていないことを表す内部状態「未設定」を区別する。「未設定」は選択肢としてUIには表示せず、「就業終了時チェック」が判定対象とする状態を指す。ユーザーが明示的に「未着手」を選択した場合はチェック対象外(=入力済み)として扱う。

### 4.1 Todo入力機能

**要件**: 就業開始前に、本日実施するTodoを入力できる。

| ID | Given(前提) | When(操作) | Then(期待結果) |
|---|---|---|---|
| AC-1.1 | 当日のTodoが0件、Todo入力欄が空 | 入力欄に「資料作成」と入力し、登録ボタンを押下する | Todo一覧に「資料作成」が1件追加され、表示されるステータスは「未設定」である |
| AC-1.2 | 当日のTodoが1件(「資料作成」)登録済み | 入力欄に「会議準備」と入力し、登録ボタンを押下する | Todo一覧の件数が2件になり、「資料作成」「会議準備」の両方が表示される |
| AC-1.3 | Todoを1件も登録していない状態からスタート | 入力欄に「A」「B」「C」を1件ずつ順に登録する | Todo一覧の件数が3件になる(登録操作の回数と一覧件数が一致する) |
| AC-1.4 | Todoを1件(「資料作成」)登録済み | ブラウザをリロード(F5)する | リロード後もTodo一覧に「資料作成」が表示され、件数は1件のままである |
| AC-1.5 | Todo入力欄が空欄 | 入力欄に何も入力せず登録ボタンを押下する | Todoは追加されず(一覧件数は変化せず)、エラーメッセージ「Todoを入力してください」が表示される |
| AC-1.6 | Todo入力欄に半角スペース3文字のみ入力されている | 登録ボタンを押下する | Todoは追加されず、AC-1.5と同じエラーメッセージが表示される |
| AC-1.7 | 前日にTodoを1件登録した状態で、端末のローカル日付が0:00を過ぎて翌日になる | 翌日にアプリを開く | 当日のTodo一覧は0件から始まり、前日のTodoは当日の一覧には表示されない(前日Todoの繰越・保持方法は6章参照) |

### 4.2 就業開始前通知機能

**要件**: 就業開始前に、Chromeで通知する。

**通知文言(正確な文言)**: タイトル「Todo入力のお知らせ」/ 本文「就業開始時刻になりました。本日のTodoを入力してください。」

| ID | Given(前提) | When(操作・イベント) | Then(期待結果) |
|---|---|---|---|
| AC-2.1 | 就業開始時刻が09:00に設定済み、Chrome通知許可が「許可(granted)」 | システム時刻が09:00になる | タイトル「Todo入力のお知らせ」、本文「就業開始時刻になりました。本日のTodoを入力してください。」のChrome通知が1件表示される |
| AC-2.2 | 就業開始時刻が09:00に設定済み | システム時刻が08:59である | Chrome通知は表示されない |
| AC-2.3 | AC-2.1で通知が表示されている状態 | 通知をクリックする | アプリのTodo入力画面を表示するタブがフォーカス(アクティブ化)される |
| AC-2.4 | Chrome通知許可が「未確認(default)」である(過去に許可/拒否のいずれもしていない) | アプリのトップ画面を開く | ブラウザの通知許可リクエストダイアログが表示される |
| AC-2.5 | Chrome通知許可が「拒否(denied)」 | システム時刻が設定済みの就業開始時刻になる | Chrome通知は表示されず、アプリ画面上に「通知が無効です。ブラウザの設定で通知を許可してください」という案内が表示される |
| AC-2.6 | 就業開始時刻が未設定 | システム時刻がどの値であっても | 就業開始通知は一度も発火しない |
| AC-2.7 | 就業開始時刻09:00の通知が本日すでに1回表示済みである | システム時刻が09:01になる、またはページを再読み込みする | 就業開始通知は再表示されない(同日中は1回のみ) |
| AC-2.8 | 就業開始時刻が09:00に設定済みで、時刻になる前にTodoが2件登録済みである | システム時刻が09:00になる | Todoの登録有無に関わらず、AC-2.1と同じ就業開始通知が表示される |

### 4.3 ステータス入力機能

**要件**: 入力したTodoに対して、未着手・完了・継続を入力する。

| ID | Given(前提) | When(操作) | Then(期待結果) |
|---|---|---|---|
| AC-3.1 | ステータス「未設定」のTodoが1件ある | そのTodoに対し「完了」を選択する | 画面上の当該Todoのステータス表示が「完了」になる |
| AC-3.2 | ステータスを「完了」に変更したTodoがある | ブラウザをリロードする | リロード後もステータスは「完了」のまま表示される |
| AC-3.3 | ステータス「未着手」のTodoが1件ある | 同じTodoに対して「完了」を選択する | ステータスは「完了」の1値のみになり、「未着手」は選択状態でなくなる(2値が同時に選択された状態にならない) |
| AC-3.4 | ステータス「完了」のTodoが1件ある | 同じTodoに対して「継続」を選択する | ステータスは「継続」に更新される(「完了」から他ステータスへの再変更が成功する) |
| AC-3.5 | ステータス「未設定」のTodoが1件ある | 「未着手」を選択する | ステータス表示が「未着手」になり、後述4.4の就業終了チェックにおいて「入力済み」として扱われる(未設定件数にカウントされない) |

### 4.4 就業終了時チェック・通知機能

**要件**: 就業終了時に、Todoに対して上記のステータスが入っているかチェックし、通知する。

**通知文言テンプレート**: タイトル「Todoステータス確認」/ 本文「ステータス未設定のTodoが{n}件あります。確認してください。」(`{n}` は未設定件数の数字に置き換える)

| ID | Given(前提) | When(操作・イベント) | Then(期待結果) |
|---|---|---|---|
| AC-4.1 | 就業終了時刻が18:00に設定済み。当日Todoが3件あり、ステータスは「完了」「継続」「未設定」が各1件 | システム時刻が18:00になる | タイトル「Todoステータス確認」、本文「ステータス未設定のTodoが1件あります。確認してください。」のChrome通知が1件表示される |
| AC-4.2 | 就業終了時刻が18:00に設定済み。当日Todoが2件あり、いずれも「完了」または「継続」または「未着手」(いずれも未設定ではない) | システム時刻が18:00になる | 未設定に関するChrome通知は表示されない |
| AC-4.3 | 就業終了時刻が18:00に設定済み。当日Todoが0件 | システム時刻が18:00になる | 未設定通知は表示されず、「本日のTodoが登録されていません」という案内が表示される |
| AC-4.4 | AC-4.1の通知が表示されている状態 | 通知をクリックする | Todo一覧画面がフォーカスされ、ステータス「未設定」のTodo(該当1件)に強調表示(例: 背景色の変化)が付与される |
| AC-4.5 | 就業終了時刻が未設定 | システム時刻がどの値であっても | 就業終了チェック・通知は一度も発火しない |
| AC-4.6 | 就業終了時刻が18:00に設定済み。当日Todoが5件あり、うち3件が「未設定」 | システム時刻が18:00になる | 通知本文が「ステータス未設定のTodoが3件あります。確認してください。」になる |
| AC-4.7 | 就業終了時刻18:00の未設定通知が本日すでに1回表示済みである | システム時刻が18:01になる、またはページを再読み込みする | 就業終了通知は再表示されない(同日中は1回のみ) |

## 5. 異常系

| No. | シナリオ | 想定される挙動 |
|---|---|---|
| 1 | Chrome通知の許可が拒否されている | 通知は表示できない。アプリ画面上に「通知が無効です。ブラウザの設定で通知を許可してください」という案内を表示する(AC-2.5) |
| 2 | 通知タイミングでブラウザ/アプリタブが閉じている、またはPCがスリープ状態である | Notification APIの制約上、通知は配信されない。次回アプリを開いた際の代替表示・再通知は行わない(3.2により対象外) |
| 3 | Todo未入力のまま就業開始時刻を過ぎた | 就業終了時チェックの対象Todoが0件となる。0件の場合は異常として通知せず、「本日のTodoが登録されていません」等の案内にとどめる(AC-4.3) |
| 4 | Todo入力欄が空、または空白文字のみで登録しようとした | 登録を拒否し、入力エラーメッセージを表示する(AC-1.5, AC-1.6) |
| 5 | 就業開始時刻・終了時刻が未設定 | 通知機能は発火しない。設定画面等で時刻未設定であることを明示し、設定を促す(AC-2.6, AC-4.5) |
| 6 | 就業終了時刻が就業開始時刻以前(同時刻を含む)に設定されている | 設定時にバリデーションエラーとし、保存を拒否する。エラーメッセージ「就業終了時刻は就業開始時刻より後に設定してください」を表示する |
| 7 | 同一Todoに対して短時間に連続してステータス変更操作が行われる(例: 未着手→完了→継続と素早く連続操作) | UI上で最後に確定した操作の値のみを最終的なステータスとして保存する。操作は発生順に逐次反映し、後から発生した操作が先行する操作の結果を上書きする(古い操作が新しい操作を上書きしない) |
| 8 | 就業終了チェックの実行中に新規Todoが追加された(Todoの削除・編集機能は3.2により対象外) | チェック処理は実行開始時点のTodo一覧のスナップショットに基づき判定する。チェック中に追加されたTodoは当該回のチェック対象に含めない |
| 9 | 日付をまたいで前日のTodoが残存している | 1.1の日付境界の定義に従い、当日分と前日以前分を区別する(AC-1.7)。前日Todoの繰越/アーカイブの要否は6章の未確定事項を参照 |
| 10 | 端末の時刻設定がずれている/タイムゾーンが異なる | ブラウザのローカル時刻を基準とする。サーバー時刻との差異が生じる場合の補正は本仕様のスコープ外とする |
| 11 | ネットワーク切断時にTodoやステータスを操作した | 操作をローカルに保持し、再接続時に同期する、またはエラーを表示し再試行を促す(実装方式は6章の保存先の決定後に別途技術検討) |
| 12 | 同一アプリを複数タブ/複数ウィンドウで同時に開いている | 就業開始/終了時刻になった際、開いているタブの数だけChrome通知が重複表示される可能性がある。重複排除は本仕様の対象外とする(3.2参照) |

## 6. 未確定事項(要確認)

以下は要件から一意に定まらないため、実装着手前に利用者へ確認が必要な事項として明記する。

- 就業開始/終了時刻の設定方法(固定値か、画面から都度変更可能か)
- 「継続」ステータスのTodoが翌日以降どう扱われるか(自動繰越の要否。1.1により当日Todo一覧は日付が変わると0件から始まる前提だが、「継続」を選んだTodoの引き継ぎ方法は未確定)
- Todoデータの保存先(ブラウザローカルストレージのみか、サーバー/DBに保存するか)
- 週末・休日の扱い(通知を止めるか等)
